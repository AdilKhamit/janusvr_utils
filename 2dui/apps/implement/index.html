• * But Dizzket... a 2D interface?!
    The idea behind this interface is to give content developers, those of which who spend most of their time multitasking, an easy desktop interface for Janus, as well as a fresh coat of paint.
  • * What's wrong with the current interface?
    There's nothing wrong with it for VR mode, but it's made for consuming content, and assumes you're wearing an oculus rift. This UI was born out of the worldbuilding team's needs. We've been needing a 'desktop friendly' mode for a long time, considering how often we jump between blender, photoshop, slack, mumble, illustrator, sketchup, etc. We can have both.
  • * I'm a busy guy... How do we get this in so I can forget about it?
    This whole thing is written in html, css and js. Just plop a transparent webkit window over Janus, add some UI javascript functions for when you click on buttons (mentioned in the Javascript tab) and it's pretty much done. By the way, this would instantly port over to mobile and a Janus web client!
  • * Give me one more good reason to throw this in or you're fired.
    Slow down, slow down! Alright, here's one... creating a really good 3D interface framework will take us a long time. The 3D interface should take advantage of things specific to VR, and not try to be some sort of frankenstein that makes sacrifices in the name of compatibility. All windows are modular components which we can reuse later, and it'll be fast as hell to prototype new systems and features (like inventory) via webkit. It'll also give our more abstract features, like the G key, a place to exist for the people that really need it.
  • * tl;dr
    Devs need more efficiency on the 2D. Throw this on a transparent webkit over Janus and it's in.
  • * uh what javascript stuff do you need then
    Oh, uh... so we could have a 'janus' scope in the webkit javascript that only the ui could access, with stuff like the following...

    * Functions?
    *janus.uifocus(boolean) - Whether or not the user is using the UI, or walking around in Janus. ESC would release mouse focus, and clicking an empty portion of the interface would allow the user to walk around again.

    *janus.navback() - Teleports to the next page.

    *janus.navforward() - Teleports to the last page.

    *janus.navhome() - Teleports to the homepage sans portal. In the future, the pocket space.

    *janus.goto(url,portal) - Teleports to the argument "url". If portal is 0, no portal spawns, and 1 spawns a portal.

    *janus.saveroom() - Saves the room.

    *janus.quit() - Quits.

    *janus.reset() - Resets.

    * Properties?
    *janus.networkstatus - Returns the current status, 0 for offline, 1 for connecting, 2 for online and 3 for error.

    *janus.networkerror - If janus.networkstatus returns a 3, returns varying error states, from name taken to whatever.

    *room.locked - Not within janus scope. Returns whether or not the room is locked.

    *room.tricount - Not within janus scope. Returns triangle count in the room

    *janus.chatlog - An array of chatlog messages. Not in room scope since you can receive messages from any subscribed room.

    *room.loadpercentage - A floating point number from 0 to 1 describing what percentage the room has finished loading.

    * Temporary functions?
    The windowing system probably won't make it into v1, so the buttons at the bottom of the screen will just open branches of the esc menu for now.

    *janus.openbookmarks() - Basically what it sounds like. Opens the 3d bookmarks menu.

    *janus.opensettings()

    *janus.openroomcodeeditor()

    *janus.openbrowser()

    *janus.openavatarmenu()

    *janus.openchatlog() - I might actually code in the chatlog for v1, so this is a 'just in case' thing.

    And that's it!
